# CLAUDE.md

## 第一性原理

以下原则优先于所有具体规则。当规则冲突时，按编号靠前的原则裁决。

### 原则 1：修改前必须穷尽全局影响

任何对共享字段、枚举、函数签名、存储格式的修改，在动手之前必须 `grep` 全仓找到**所有**引用点。审查报告给的代码位置是入口线索，不是完整清单。

> 反面案例：7 轮修复中多次出现「修了 A 文件的字段名，但 B/C/D 文件的同名引用未同步」，根源就是只按报告给出的行号修改，没有自己做全仓搜索。

### 原则 2：数据完整性优先于功能完整

静默数据丢失、覆盖写入、统计归零、状态不一致是 P0 级问题。功能缺失（少一个表单字段、少一个导出格式）是 P1/P2 级问题。

- 做减法（删除旧代码/旧字段）时：确保无残留引用再删
- 做加法（新增写入路径）时：确保失败有回滚、成功有验证、UI 有反馈
- 做修改（重命名字段）时：确保新旧字段有兼容期，或全量迁移

> 反面案例：「补全表单缺 PRD 成本字段」是 P1，但「saveEntry 用未定义变量保存导致 Runnable Error 崩溃」是 P0。应先保证不崩溃、不丢数据，再扩展表单字段。

### 原则 3：新增链路必须端到端推演

每新增一条用户可感知的数据流（如「点击保存 → 生成 Entry → 今日统计刷新」），必须在提交前从起点到终点逐层推演：

1. 构造层：数据以什么类型/格式写入？
2. 传输层：消息/回调携带了哪些字段？字段类型是什么？
3. 持久化层：写入的 key 和格式与查询条件是否一致？（ISO 字符串 vs 数字时间戳）
4. 读取层：所有消费方能否正确解析新格式？
5. UI 层：显示的数据是否来自正确的字段？空值/异常时是否有降级展示？

> 反面案例：Popup 补全表单保存路径未检查 `SourcePlatform` 是否加载、`getStatus` 是否返回 `toolId`、`getEntriesByDate` 能否查到 ISO 格式的 `createdAt`。单一功能涉及 5 层调用，但只验证了其中 2 层。

### 原则 4：契约优于实现

`shared/schemas/` 中的 JSON Schema 是跨端数据契约。任何一端的数据模型偏差（字段名、枚举值、单位、类型）必须在提交前与 schema 对齐。实现可以分阶段（原型用 JSON、正式用 SQLite），但数据形状不能分阶段。

- 平台枚举：`macos / android / browser`，不允许多端各自定义
- 状态枚举：`pending / completed / ignored / merged / suspected`，不允许多端各自扩展
- 字段单位：时间用分钟（`estimated_saved_minutes`），不允许多端各自用秒/小时/比例

> 反面案例：Android 用 `MOBILE_APP / FOREGROUND_APP`、浏览器扩展用 `browser-chatgpt / browser-deepseek`。三端数据导出的 CSV 无法合并、周报口径不一致。

### 原则 5：提交前以审查者视角自审

每次修改完成后、提交修复报告前，用审查者视角对自己的 diff 做一轮独立审查：

- **同类 grep**：改了 `getSessionsByDate` 的日期逻辑，全仓还有多少地方用同样的日期处理？
- **全链路推演**：改了写入，所有读取方跟进了吗？
- **失败路径**：try-catch 是吞掉了还是抛出了？UI 是否感知失败？
- **边界条件**：空值、null、undefined、跨午夜、跨时区是否覆盖？

> 反面案例：7 轮修复的根因不是技术能力不足，而是每次修完后没有做「如果我是审查者，我会找什么」的自检。

## 修改纪律

以下规则是第一性原则在具体场景中的落地。

### 规则 1：改字段/枚举前必须先全局 grep，改后验证零残留

修改任何共享字段名、枚举值或函数签名时：
- **修改前**：`grep` 所有引用旧值的位置，建立完整影响清单
- **修改后**：再次 `grep` 确认旧值零残留

这是原则 1（穷尽全局影响）和原则 4（契约优于实现）的直接应用。

### 规则 2：Schema 迁移必须同步写入侧和所有读取侧

数据字段重命名（如 `startTime → startedAt`）必须同时更新：
- 写入侧（构造该对象的代码）
- **所有**读取侧：storage 查询、API 响应组装、UI 展示、计算器、导出、周报
- 检查清单：`grep` 旧字段名全仓，逐项确认

这是原则 1（穷尽全局影响）和原则 3（端到端推演）的直接应用。

### 规则 3：新功能做完后必须全链路推演

每新增一条用户可见的交互路径（如「点击保存 → 生成 Entry → 刷新统计」）：
- 从用户操作起点到 UI 刷新终点，逐个函数校验输入/输出类型是否匹配
- 特别检查：ISO 字符串 vs epoch 数字、字段名是否被消费方引用、依赖模块是否已加载

这是原则 3（端到端推演）和原则 5（审查者自审）的直接应用。

### 规则 4：原子性问题要改操作顺序，不是改调用方式

当 A → B 两步存在「A 成功但 B 失败导致数据不一致」风险时：
- 不要只是把两次调用合并到一个函数里（仍然是 A → B）
- 应该反转顺序为「先做不可回滚的（B），成功后做可补做的（A）」
- 或者改为「暂存状态 → 两步都成功 → 提交」

这是原则 2（数据完整性优先）的直接应用。

### 规则 5：修 bug 后自查同类问题再提交

提交修复报告前，对自己刚改的 diff 执行一遍审查者视角的自查：
- 同类问题 grep：你改了 `getSessionsByDate`，那 `getEntriesByDate`、`crossesMidnight`、周报过滤有没有同样的问题？
- 字段迁移消费方：你改了 sessionizer 写入侧，storage/service-worker/popup/options/calculators 读到了吗？
- 异常处理：你加了 try-catch，失败路径的 UI 反馈和可恢复性考虑了吗？

这是原则 5（审查者自审）的直接应用。

### 规则 6：优先保证「不写坏数据」，再追求功能完整

数据一致性问题（静默覆盖、非原子写入、状态不一致）的优先级高于 UI 字段多寡。宁可少一个表单字段，不能让用户保存后数据丢失或统计归零。

这是原则 2（数据完整性优先）的直接应用。
